Surfaced as an inline `AddressEditor` on EVERY kid in the Pickup Map side panel (pinned + all 3 flagged buckets). Caveat: it geocodes WITHOUT any existing postal_code (ignores it).

**FINDING A (2026-06-29 — wrong-van, NOT no-van): no max-distance sanity guard.** `nearestStopId` (`route-build.ts:21`) always returns SOME zone; `autoAssignStopsFromAddresses` flags ONLY kids with no point (`routing.ts:166`). A kid whose address geocodes to a far-but-valid wrong location (live Sisseton case: ~11 kids geocode 148–390mi away from misspelled towns, e.g. "Barker hill"→Berkery Hill) gets coords → `geocodeFailed=false` (`assign/page.tsx:137`) → assigned to the NEAREST zone (wrong van), shown as a normal colored pin, in NO flag list; the Suggest toast counts them as "located/assigned" success. Only cue = map `fitBounds` zooming way out for the far pin (which also collapses local homes into an unreadable cluster). No-child-DROPPED invariant holds; no-child-on-WRONG-van does NOT. Fix: distance threshold in autoAssign (reuse `haversineMeters`) → flag "home is N mi from its van — check the address" instead of silently assigning, or show per-kid distance-to-assigned-van.

**FINDING B (2026-06-29 — Suggest is one-way fill; can't self-correct after an address fix).** `assignStopsForMode` fills only EMPTY legs (`route-build.ts:67`; van mode anchors the empty leg to an already-pinned leg, lines 59–64). The FIRST Suggest click geocodes AND assigns in one pass (`routing.ts:127–189`), so the far-geocoded kids get wrong-van legs immediately. Fixing the address inline + re-running Suggest does NOT move them (legs no longer empty → no override). To correct an already-assigned kid you MUST use manual "Put on [van]" (`assignStudentToVan`→`assignLegsForVan`, which overrides). So "does fixing a bad address move the kid after Suggest?" → only if not yet Suggested/assigned. On a clean slate: fix bad addresses BEFORE the first Suggest, or use manual reassign after.

**GAP 3: geocoding never auto-runs** — lat/lng stay NULL until someone clicks "Build from addresses." Safe (kids stay flagged) but easy to forget. Consider auto-prompt or worklist copy.

**GAP 4 (THE go-live gate item, STILL OPEN as of 2026-06-18 final readiness review): Nominatim throttle + silent failure conflation.** `routing.ts:104-106` fires `GEOCODE_BATCH=8` via `Promise.all` with no delay (grep-confirmed: zero sleep/throttle/setTimeout in geocode.ts or routing.ts). `MAPBOX_TOKEN` is `.optional()` (`env.ts:23`); when unset on prod the live path is Nominatim, whose policy is strictly 1 req/sec, no bursting → 8-wide bursts get 403/429/temp-ban. TWO problems: (a) the burst itself; (b) `geocodeNominatim` swallows every non-OK as `null` (`geocode.ts:55`), so a throttled/errored family is counted as `flagged` — INDISTINGUISHABLE from a genuinely address-less kid. On ~100 families that silently flags a dozen+ valid-address kids as "no address." No child is DROPPED (they hit the worklist → invariant holds), but the one-click build looks broken on the free path and the coordinator gets no rate-limit signal. **Go-live fix (pick one): (1) set MAPBOX_TOKEN on Vercel prod — cleanest, batch-8 is fine there [PREFERRED gate]; or (2) serialize the Nominatim branch ~1.1s apart (GEOCODE_BATCH=1 for the no-token path) + add an `email=`/contact param + report geocode-FAILURE distinctly from no-address.** The `GEOCODE_BATCH=8` "Mapbox is fast" comment is wrong for the fallback.

**GAP 7 (nit): geocode.ts has ZERO tests + nearestStopId tie-order undefined.** `geocode.ts` does two-provider response parsing (Mapbox `center` is `[lng,lat]` — swap is easy to get backwards; Nominatim field is `lon` not `lng`) + fallback selection, all unit-testable with a stubbed fetch — none tested. Separately, `nearestStopId` uses strict `<` so exact ties resolve by DB array order (stops query has no `.order`) → non-deterministic, contradicts "deterministic route-build." Add `.order("id")` + a tie test.

**~~GAP 5: "assigned" ≠ "on a van"~~ — CLOSED (verified 2026-06-29).** `autoAssignStopsFromAddresses` now filters candidate stops to (on a van route) AND (has coords) before nearest-zone (`routing.ts:70–98`), and errors out if no van has a routed+located zone. So a Suggest assignment always lands on a real van.

**GAP 4 UPDATE (2026-06-29): failure now distinguished ON THE MAP.** Migration 0027 added `families.geocode_failed_at`; a tried-but-unmatched address shows in a distinct rose "Address didn't match" bucket vs. amber "Needs an address" vs. sky "Not on the map yet" (`pickup-map.tsx` NoAddressList). The autoAssign `flagged` COUNT still lumps failed+no-address, but the coordinator can tell them apart on the map. Geocoding is now Sisseton-biased (Mapbox `proximity` + Nominatim `viewbox`/`countrycodes=us`, `geocode.ts:13–18`) — reduces but does not eliminate far mis-geocodes (Finding A). Throttle gate still applies only on the no-Mapbox-token path.

**GAP 6 (confirmed, cross-owned): ETA is crow-flies haversine.** `src/lib/geo.ts estimatedEtaSeconds` = haversine ×1.4 ÷11 m/s, wired only into the parent page, not the map. Real road ETA needs Mapbox Directions Matrix. Fallback already haversine.
