The **Pickup Map** at `/coordinator/vans/assign` is the coordinator's manual door-to-door routing tool, built 2026-06-27. A Leaflet/OSM map shows one pin per van-needing kid's HOME for the day; each pin is the kid's NAME + a checkbox. Pin color = the van's pickup-zone color the kid is currently on (`morning_van_id ?? afternoon_van_id` → that van's zone stop `color_code`); GREY (`#9ca3af`) = no van yet. Coordinator checks kids (pin or mirrored side list), picks a target van (color-chip buttons), clicks "Put N kids on [Van]".

**Why:** the user's headline feature — auto-suggest is the first pass, this is the human override surface the non-negotiable rules require (assignment must be reviewable + adjustable before use).

**How to apply:**
- Route is `/coordinator/vans/assign` — NOT `/coordinator/vans/map` (that's the live GPS tracking map). Nav label "Pickup Map" vs "Live Map".
- Assignment reuses `assignStudentToVan` (one call per selected kid via `Promise.allSettled`; per-kid failures surfaced as toasts — e.g. a boarded kid can't be moved, the `boardedStopConflict` guard). It does NOT write stop legs directly.
- **PER-DAY vs ALL-DAYS split (verified 2026-06-29 — the cause of "Day 1 routed, Days 2/3 short"):** the prominent flow (select pins → bottom bar "Put N on van", `runAssign`) is **single-day** — `assignStudentToVan({ eventDate: date })` (`pickup-map.tsx:196`). The geocoded `SideList` rows have NO all-days control. The **all-days** action `assignStudentToVanAllDays` (`students.ts:502`, loops all `VBS_DATES`) is wired ONLY into `RegionAssign`, which appears ONLY on the flagged no-address buckets. So a coordinator routing located kids on the map naturally produces Day-1-only assignments. **`autoAssignStopsFromAddresses` ("Suggest") ignores its `eventDate` arg and routes ALL VBS days in one pass** (`routing.ts:108-119`) — so re-running Suggest backfills the later days (caveat: it assigns NEAREST zone, so it diverges any deliberate non-nearest Day-1 override; copy-Day-1 stops for those).
- **CROSS-DAY BLIND SPOT (blocker class):** both the dashboard (`coordinator/page.tsx:63`) and this Pickup Map (`assign/page.tsx:56`) default to ONE day via `defaultVbsDate` (clamps pre-event → first day). A kid routed Day 1 but not Days 2/3 shows fully-routed on every DEFAULT surface; the shortfall is only visible by manually setting `?date=` per day. No multi-day routed-count / "routed today not tomorrow" signal exists. Verify each VBS day's needs-routing separately.
- "Suggest from addresses" = `autoAssignStopsFromAddresses` (the existing nearest-van first pass). "Locate N homes" = new `locateStudentHomes({ eventDate })` in `src/server-actions/routing.ts` — geocode-ONLY (writes `families.lat/lng`, no stop assignment) so pins appear before manual assignment; returns `{ located, stillMissing }`, capped ~75/run.
- Un-pinnable kids are split into THREE labeled buckets (`pickup-map.tsx` NoAddressList, verified 2026-06-29): rose "⚠ Address didn't match" (has address, `geocode_failed_at` set, lat null), sky "📍 Not on the map yet" (has address, not located), amber "⚠ Needs an address" (no street address). Driven by `families.geocode_failed_at` (migration 0027). Never silently dropped.
- **Inline `AddressEditor`** on every kid (pinned + all 3 buckets) → `setStudentHomeAddress({studentId, streetAddress, city})` in `routing.ts`: updates the FAMILY (all siblings), hard-sets state "SD", geocodes on the spot, sets lat/lng+clears failed flag on success / sets failed flag on fail, then `router.refresh()`. KNOWN GAP: after fixing a bad address, "Suggest" will NOT move a kid who was already assigned (Suggest only fills empty legs) — must re-assign via "Put on [van]". **Distance guard NOW EXISTS (verified 2026-06-29, supersedes the old "no distance guard" note):** `autoAssignStopsFromAddresses` flags any home >`MAX_HOME_TO_ZONE_MILES` (40mi) from every zone — clears coords + sets `geocode_failed_at` → surfaces under "Address didn't match", never a silent wrong-van pin (`routing.ts:183-198`). BUT the guard runs ONLY in Suggest, not in `setStudentHomeAddress`/`locateStudentHomes`, and a <40mi mis-geocode can still auto-assign a plausible-but-wrong van via Suggest. `localPlace` (`geocode.ts:38-50`) maps 4 local town names (Barker Hill / Long Hollows / Old Agency / Peever Flat) to hard-coded centers decoupled from van-zone coords → nearest-zone picks whichever van is closest to that center; reviewable as a colored pin, not silent.
- Geocoding is Sisseton-biased (`geocode.ts`: Mapbox `proximity`, Nominatim `viewbox`+`countrycodes=us`). Siblings at one coordinate fan out via `buildStackIndex` (stacked-marker offset) so names don't overlap.
- Pure partition/color logic is `src/lib/van-assign-map.ts` (`buildVanAssignMapData`, `vanColor`, `UNASSIGNED_PIN_COLOR`), tested in `tests/unit/van-assign-map.test.ts`. Leaflet rendering itself isn't unit-tested (consistent with the live map). Geocoding hits a live API and is not exercised by tests.
- Families coord columns: `families.lat` / `families.lng` (double precision); address parts `street_address/city/state/postal_code`.
