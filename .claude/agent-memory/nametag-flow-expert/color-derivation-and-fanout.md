The day color chain: `student_day_status` view (live version = migration **0023**, redefined by `create or replace`; latest wins) computes `wristband_color_for_day = coalesce(s_pm.color_code, s_am.color_code)` and `wristband_color_name = coalesce(s_pm.color_name, s_am.color_name)`. The view joins `s_am`/`s_pm` internally but still exposes only the single coalesced color + name (not the individual leg colors as columns).

**Two-color tags SHIPPED (verified 2026-06-17):** the nametag flow gets per-leg colors WITHOUT a view change — `nametags/page.tsx` loads `stops` (id, name, town, color_code, color_name) into a `stopMap`, and `buildTagData` looks up `morningStopId`/`afternoonStopId` against it to populate `morningColorCode/Name` + `afternoonColorCode/Name` on each `NameTag`. The coalesced `colorCode`/`colorName` is still carried for the single-band path AND for `sortTags` grouping. `NameTagBand` in `nametag-sheet.tsx` renders a split AM|PM band only when `am && pm && am !== pm`; otherwise the single coalesced band. So the view did NOT need to expose leg colors — the page already had the stop rows. (Earlier note that this was data-layer-blocked is obsolete.)

**Fan-out consumers of `wristband_color_for_day`/`_name`** (grep-verified 2026-06-16): coordinator dashboard (`/coordinator`), `/coordinator/print` failsafe, `/coordinator/students` roster, `/coordinator/nametags`, `/parent/[familyToken]`, `/van/[vanId]`, and `server-actions/events.ts`.

**`updateStopColor` revalidation gap:** it `revalidatePath`s `/coordinator`, `/table`, `/van` (layout scope) but **NOT `/parent`** — a color edit looks applied everywhere except a stale parent page until it re-renders. The parent page does read both color columns. See [[print-fidelity-and-failsafe-status]].

**TWO color-edit doors to the same `stops` rows (verified 2026-06-18 review).** Under door-to-door, zone color is set primarily on the **van-manage** screen: `provisionVanZone` (`server-actions/vans.ts` ~line 113) writes `color_code`/`color_name` at van creation, and `updateVan` (~line 182) edits `color_code` on the van's zone stop. The OLD `/coordinator/stops` page + `updateStopColor` still works (edits the same rows, still nav-linked as "Colors") but is now redundant, has stale "Each town has a color" framing (`stops/page.tsx:38` — under door-to-door a stop IS a van zone / `town == vanName`), and its revalidation set DIFFERS from the van path: `updateStopColor` → `/coordinator`+`/table`+`/van`; `updateVan`/`revalidateVanTrees` → `/coordinator`+`/van` only (no `/table`). Neither hits `/parent`. **Recommendation (not yet actioned — user decision):** retire `/coordinator/stops` + `stop-color-editor.tsx` + the "Colors" nav link and recolor only via van-manage (color lives with the van; van-manage's color field is also the address→van geocode anchor). **Why:** two doors to the same data = more last-write-wins races + divergent fan-out. **How to apply:** if asked to "recolor a van zone," prefer the van-manage path; flag the redundancy if touching either.

**Default date is VBS-window-clamped (since 2026-06-18, commit 4a2b64c).** `nametags/page.tsx` defaults `day` to `defaultVbsDate(getLocalDate())` (in `src/lib/registration/dates.ts`), NOT raw today — before the event it shows day 1 (2026-06-22), after it shows the last day, during it shows today. `?date=` still overrides. Shared with the coordinator dashboard + groups. **Why:** day-records only exist for VBS week, so raw "today" showed an empty roster pre-event.

**Allergy/medical removal in signup is a NON-issue for tags (audited 2026-06-18).** The caregiver-friendly signup (commit bbf1cee) combined allergies into one `medical_notes` box and writes `allergies: null`, but the `students` columns are untouched and `registerFamily` still inserts both. Tags never selected those columns anyway — the nametag student select is `id, legal_first_name, legal_last_name, preferred_first_name, wristband_code`. No tag-side dependency on anything signup stopped providing; the view color columns (`wristband_color_for_day/_name`) and per-leg stop ids are all intact.

**GO-LIVE confirmed (2026-06-18).** Full re-review for VBS week: `buildTagData`/`sortTags` correct; band precedence in `NameTagBand` is needs-routing (red) > two-color (split AM|PM) > single coalesced; color-NAME text failsafe present on every band (single shows `colorName ?? "P"`, two-color shows `AM <name>`/`PM <name>`, needs-routing is literal text); `defaultVbsDate` clamps pre-event to day 1; `updateStopColor` still coordinator-gated + hex-validated. 0023 view exposes mode/stop-ids/van-ids/color name as selected. typecheck clean, 29 nametag unit tests green. The only standing nit is the `/parent` revalidate gap below (not a tag blocker). Verdict: GO.

**NULL-stop van kids — FIXED (verified 2026-06-17).** `morning_van_id` is derived by joining `morning_stop_id` against the AM route's stop_ids, so a van kid registered with NULL stops gets `morning_van_id = NULL` and `wristband_color_for_day = NULL`. The tag flow no longer prints a misleading "P · Parent drop-off" band for these: `buildTagData` sets `needsRouting` via the shared `needsRouting()` helper in `src/lib/routing.ts` (van leg required + no van id resolved, directional by mode; parent-both/non-attending never flag). `sortTags` floats needs-routing tags FIRST, and `NameTagBand` short-circuits to a loud red "⚠ Needs routing — see coordinator" band before any color logic. The directional check (`ridesMorningVan`/`ridesAfternoonVan`) means a `parent_dropoff_only` kid only flags on a missing afternoon van, etc. This is the canonical routing definition shared with the coordinator worklist + paper failsafe.
