The coordinator live van map (`/coordinator/vans/map`) is **Leaflet 1.9.4 + OpenStreetMap raster tiles**, confirmed against `package.json` (no mapbox/maplibre dep) and code. **`MAPBOX_TOKEN` is NO LONGER dead config** (as of ~2026-06-16): `src/lib/geocode.ts` uses it for address geocoding (Mapbox when set, else Nominatim fallback). It is still NOT used for map tiles or for ETAs. CLAUDE.md's "Mapbox tiles / Directions Matrix ETAs" notes remain stale for the *map* — the map is OSM, ETAs are haversine.

**Why:** Future map work must not assume Mapbox tiles/styles exist, but also must not call `MAPBOX_TOKEN` "dead" — it's live for geocoding (owned by the location/routing agent).

**How to apply:** Tiles come from `{s}.tile.openstreetmap.org` (external CDN runtime dep; OSM policy discourages app traffic — self-hosted/commercial tiles are the real fix if traffic grows). **Marker images from `unpkg.com` are GONE:** the old `L.Icon.Default.mergeOptions({...unpkg...})` block in `van-map.tsx` was removed 2026-06-16 — it was dead (stops are `circleMarker`, vans are `divIcon`+`L.marker` with an explicit icon; nothing used the default icon). Don't reintroduce a default-icon CDN dep unless a marker actually needs it.

## Realtime / marker mechanics (verified)
- `van-map.tsx` subscribes to its OWN channel `van-locations-map` on `postgres_changes` for `public.van_locations`, event `*`. On each payload it replaces that van's row in `locations` state by `van_id`, which re-runs `renderVanMarkers` to move the marker in place (upsert-by-id Map of Leaflet markers). DELETE won't remove a marker via this path (rows are upserted, low impact), and `payload.new` is `{}` on DELETE so `row.van_id` is undefined → the `if (!row) return` guard handles it but a true delete leaves a stale marker.
- `van_locations` has `replica identity full` and is in the `supabase_realtime` publication (migration 0007), so realtime payloads carry full rows — the subscription's reliance on `payload.new` fields (lat/lng/accuracy_m/reported_at) is sound.
- **Double realtime work on this route:** the coordinator layout mounts `RealtimeRefresher` (default tables include `van_locations`) which debounced-`router.refresh()`es on every GPS change, AND the map's own channel moves the marker. Redundant but functionally fine. The server-component re-fetch from `router.refresh()` re-seeds `initialLocations`, but the client `locations` state is the source of truth for markers (state isn't reset by the refresh), so they don't fight.
- Map init/cleanup is correct: `map.remove()` on unmount, channel `removeChannel` on unmount, 15s re-tick interval cleared.

## The WHOLE map is RLS-gated — it only works for a signed-in staff session (verified 2026-06-29, pre-event)
- Every read the map needs is gated on a real Supabase auth session: `stops_select_all`/`vans_select_all` require `auth.uid() is not null`, and `van_loc_staff_select` requires `_is_staff()` (→ `_current_role()` → `select role from public.users where id = auth.uid()`), all in `0006_rls_policies.sql`. `map/page.tsx` fetches with the **cookie-bound** server client (`@/lib/supabase/server`, NOT admin) — so no RLS bypass on the initial render either.
- **Kiosk / no-login mode (`ALLOW_NO_LOGIN=true`) produces NO browser Supabase session.** `getSessionUser` resolves a synthetic coordinator via the *admin* client but never establishes browser auth cookies. So in kiosk mode: `auth.uid()` is null everywhere → initial server fetch returns empty vans/stops/locations (blank map) AND the browser realtime channel connects as anon → **postgres_changes for `van_locations` are never delivered → markers never move.** Realtime is ALWAYS browser-RLS-gated (the service key can't reach the browser), so even if the server seeded initial positions, live movement would still be dead in kiosk mode.
- **Net:** the live van map is functional ONLY when the coordinator is actually signed in (email+password → real JWT in cookies → `createBrowserClient` carries it → `_is_staff()` true → realtime delivers). The same applies to the `/coordinator` dashboard (also cookie-client), so if the dashboard shows data, you're in signed-in mode and the map's realtime works too. This is an auth/RLS-owned precondition (auth-session-expert + data-integrity), not something the map code can fix — but it's THE thing to confirm before an event if there's any chance `ALLOW_NO_LOGIN` is on.

## Freshness model (SHIPPED — verified 2026-06-16, was a gap before)
- `src/lib/gps-freshness.ts` `gpsFreshness(reportedAtMs, nowMs, opts?)` → `"fresh" | "stale" | "dark"`. Defaults **120s stale / 600s dark** (NOT the parent page's 300s — map vans report ~15s so a couple misses = stale). Future timestamps clamp to fresh. Pure + unit-tested (`tests/unit/gps-freshness.test.ts`, 5 tests).
- Wired into `van-map.tsx` three ways: marker bg color (`FRESH_MARKER_BG`: green `#15803d` / amber `#b45309` / red `#b91c1c`), the side-panel per-van line (`VanFreshness`: "● Xm ago · check" / "· not reporting"), and a panel-level "⚠ N vans not reporting" banner (counts any non-fresh van). A dark van now reads red everywhere — the green-forever safety gap is closed.
- **Device-clock nit STILL OPEN (low sev) — but the right framing is COORDINATOR clock, not aide device:** `reported_at` is stamped on the **server** inside the broadcast action (`van.ts:77` `new Date().toISOString()`, runs on Vercel = reliable UTC), NOT on the aide's device — so a flaky aide phone clock is irrelevant. Freshness compares that server stamp against the coordinator's **browser** `Date.now()` (`van-map.tsx` `now` state). So the only skew that matters is the *coordinator's own machine*. A skewed coordinator clock mis-ages the whole fleet uniformly; a *slow* coordinator clock is the unsafe direction (keeps a dark van green past 10m). Needs multi-minute skew to matter given the 120/600s thresholds. Fix only if it bites: send server `now()` in the payload and offset, or stamp freshness server-side. Verified again 2026-06-17.
- **NEVER-REPORTED van is the top remaining safety gap (not the same as "dark"):** freshness only grades vans that have *ever* upserted a `van_locations` row. A van that never started broadcasting has no row, so `VanFreshness` shows a neutral grey "no GPS" (`van-map.tsx:366`) and it's **excluded** from the "N not reporting" banner count (`van-map.tsx:247-250` requires `loc` to exist). A van that never came online reads as quietly idle, not alarming. The "dark" color-downgrade fix did NOT cover this case. Fix: cross-reference today's `van_assignments`; a van assigned-today with no/absent location is its own alert state.

## Door-to-door ripple: van pickup ZONES are now stops rows → they render as map markers (verified 2026-06-18)
- Under the door-to-door model each van owns ONE pickup-zone `stops` row (created by `provisionVanZone` in `server-actions/vans.ts`; name/town/color_name = the van's name). **There is NO discriminator column on `stops`** (no `is_zone`/`kind` — confirmed `0002_core_tables.sql:149`), so a zone stop is byte-identical in shape to a real corner stop.
- `map/page.tsx` fetches **all** `stops` and renders every one with non-null lat/lng as a colored `circleMarker`. So the map CANNOT tell a zone from a real stop.
- A new zone is inserted with **null lat/lng** (`provisionVanZone` omits them) → filtered out by `page.tsx:63` → no marker. BUT `updateVan` writes the van's "area location" lat/lng onto the zone stop (`vans.ts` `zoneUpdates.lat/lng`) the moment a coordinator sets it — and the door-to-door flow *encourages* setting it (it powers address→van matching). Once set, the zone stop gets a circle marker on the map in the van's color.
- **Net effect (low sev, cosmetic, NOT a regression that breaks anything):** the map's "stops" layer now shows van pickup-AREA centroids, not corner stops (under door-to-door there ARE no corner stops). A coordinator could be mildly confused (a static colored circle = a van's zone center, vs. the moving van marker = the van's live GPS). The two are visually distinct: stop = `circleMarker` radius-8 dot; van = `divIcon` name-label rectangle. No crash, no realtime impact, no freshness impact.
- **If this confuses coordinators, the map fix is mine:** relabel the "Stops" control → "Zones", and/or the circle popup ("<van> pickup zone"). Distinguishing zone-vs-real-stop at the data layer (a column) is the data-integrity/van-management agents' call — don't add it unilaterally. For now `fitStops`/`hasStops`/the "Stops" button still work (they just fit/show zone centroids).

## Coordinator "you are here" dot (SHIPPED 2026-06-29, commit 7d86296)
- `van-map.tsx` now runs a SECOND `watchPosition` for the COORDINATOR's own device (separate from any aide/van broadcast). It sets `me` state; a blue `circleMarker` (`#3b82f6` fill / `#1d4ed8` stroke, radius 7) is drawn/moved in a `useEffect` keyed `[me, mapReady]`; the `leafletRef` ctx gained a `meMarker` slot; a "📍 Me" button (`flyToMe`, disabled until `hasMe`) recenters. Watch errors (denied/unavailable) are swallowed → no dot. Watch is cleaned up via `clearWatch` on unmount. Verified: no regression to van markers / realtime / freshness; tsc + lint clean.
- **Non-obvious correctness detail to preserve:** the me-marker effect reads `leafletRef.current` (a non-reactive ref), so it needs a manual re-trigger for the GPS-fix-arrives-before-map-init ordering. That trigger is the new `mapReady` state: init sets `leafletRef.current` THEN `setMapReady(true)` (order matters — ref must be set first), and `mapReady` is in the effect's dep array purely to re-run it once the map exists. Don't "clean up" `mapReady` as an unused dep — removing it reintroduces the race where an early GPS fix never draws the dot. This dot does NOT participate in freshness/realtime; it's the coordinator's own browser GPS only.

## geo.ts ETA
Pure haversine × 1.4 ÷ 11 m/s (~25mph). Consumed ONLY by the parent page (PM transit, when not stale) + unit tests (`tests/unit/geo.test.ts`). The map shows NO ETA. Helper is correct for what it is; the 1.4 fudge + fixed speed can mislead on real roads. See [[freshness-and-cdn-notes]].
