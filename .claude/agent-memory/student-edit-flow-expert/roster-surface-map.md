The coordinator roster surface, mapped against actual code (verified 2026-06-16).

**Client choice per action (admin = RLS bypassed, cookie = RLS enforced):**
- `updateStudent` (students.ts) — cookie-bound (`createClient`). RLS backstop present.
- `updateStudentDayRecord` (students.ts) — cookie-bound. RLS backstop present.
- `updateTodayStops` (day-record.ts) — cookie-bound. Wired into `/table/[code]/change-stops.tsx` (ChangeStopsPanel) — NOT dead/unwired. Has both the mode/stop consistency guard AND the boarded-stop guard.
- `updateFamilyContacts`, `updateGuardianPhone`, `getFamilyAccessUrl`, `rotateFamilyToken` (families.ts) — **ADMIN client. RLS bypassed. `isCoordinator` is the only defense.** Why admin: guardians/tokens reads/writes need to cross family boundaries a coordinator's RLS would otherwise scope.
- `searchStudentsByName` (events.ts, not students.ts) — admin client, gated by `isStaff` + PostgREST filter sanitization (`q.replace(/[.,%_*()\\]/g, "")`). Both guards load-bearing; returns names+wristband codes to ANY staff role by design.
- `deleteStudent` (students.ts, WIP as of 2026-06-28) — **ADMIN client, `isCoordinator`-gated.** Admin justified: `student_day_events` is append-only with no delete RLS policy, so only the service role can clear a kid's event history. Deletes events FIRST (`student_day_events.student_id` is ON DELETE RESTRICT, 0003), then `students` (cascades day_records 0002:202 + anomaly_notifications 0016:8; nulls incidents 0002:275 SET NULL). The self-ref `superseded_by_event_id` RESTRICT (0003:30) is intra-student so the single bulk delete is safe. Two-statement, non-transactional. Confirm-gated UI (`delete-student.tsx` → ConfirmDialog), redirects to `/coordinator/students` (no 404). **Gap: no current-state guard** — deleting a currently-boarded kid strands the aide same as the boarded-stop edit; should at least warn on a live non-terminal state.
- `updateStudentPhoto` (students.ts, WIP) — ADMIN client, `isCoordinator`-gated. Writes `${family_id}/${studentId}.jpg` with `upsert:true` to private `student-photos` bucket — SAME key as registration (registration.ts:180), so a replace overwrites, never orphans. Client resizes to ≤800px JPEG (PhotoInput) → base64 → action. UI must `router.refresh()` to re-sign the URL.

**Why the boarded-kid stop edit is dangerous (DB-level confirmed):**
`student_day_status` view (0005) derives `afternoon_van_id` purely from `afternoon_stop_id` → pm_route join (line 73). `_authorize_event` (0017, lines 75-84) authorizes a driver/aide PM event by `v_afternoon_van = v_assigned_van`. So editing the afternoon stop AFTER `van_boarded_am` silently re-points the derived afternoon van; the aide holding the kid can lose authz to offload them, and the boarding event + derived van disagree.

**B1 FIXED (verified 2026-06-16):** `boardedStopConflict(state, current, next)` in `src/lib/routing.ts` (tested, 10 tests in routing.test.ts) blocks the strand on BOTH write paths — `updateTodayStops` (day-record.ts) and `updateStudentDayRecord` (students.ts). Both read `state, morning_stop_id, afternoon_stop_id` from `student_day_status` and reject changing the stop of the leg the kid is currently riding (`van_boarded_am`→morning, `van_boarded_pm`→afternoon). Directional + leg-scoped: does NOT block the other leg or pre-board (not_started/etc.) re-routes, allows no-ops. Error surfaces via `toast.error` on both UIs. NOTE: `updateStudentDayRecord` only runs the guard when a stop FIELD is in the update payload — a pure mode/attending edit skips it (correct: no stop is moving). The edit form (student-edit-form.tsx) always sends both stop values though, so a no-op there returns null. NOT app-layer-only protection, but also NOT a DB constraint — a direct `student_day_records` UPDATE (e.g. Studio) still bypasses it.

**MODE-CHANGE STRAND GUARD (added ~2026-06-17, students.ts only, UNTESTED):** `updateStudentDayRecord` now ALSO blocks a *mode* switch that takes a boarded kid off the van — `van_boarded_am` + new mode not in {van, parent_pickup_only} → reject; `van_boarded_pm` + new mode not in {van, parent_dropoff_only} → reject. PLUS an unused-leg clearing rule (lines 135-141): on a mode change, a leg the new mode doesn't use gets `stop_id = null` unless the caller explicitly set that leg this call. Both are INLINE in the action, NOT a pure helper, and have ZERO unit tests (the `ridesAm/ridesPm` matrix + the clearing rule are untested). `updateTodayStops` has no mode param so this gap is students.ts-only. RECOMMENDED fix: extract `modeStrandsBoardedKid(state, newMode)` (and optionally `clearUnusedLegStops`) into routing.ts and unit-test the matrix. The `ridesMorningVan/ridesAfternoonVan` helpers already in routing.ts (tested) duplicate the inline `ridesAm/ridesPm` derivation — a 3rd copy.

**Roster filter/sort logic extracted (verified 2026-06-17):** pure `filterStudents`/`sortStudents`/`presentStates` in `src/lib/coordinator/student-filter.ts` (tested, student-filter.test.ts); students-table.tsx just wires state to them. Filters compose AND (query + age range + status all must pass). Age range correctly EXCLUDES unknown-age kids. Two known asymmetries: (S1) the roster page maps `firstName: preferred_first_name ?? legal_first_name`, so the searchable `firstName` is the DISPLAY name — a preferred-name kid's LEGAL first name is unsearchable (last name still matches). (S2) `updateStudentDayRecord`'s zod mode↔stop consistency check only fires when `mode` is in the payload (`if (!data.mode) return`); its sibling `updateTodayStops` checks against the STORED mode regardless. UI always sends mode so not reachable from the form, but the two public actions disagree on the invariant.

**Age tooling status (built 2026-06-16, verified):** age filter on `/coordinator/students` (per-age chips + min/max range) and `/coordinator/groups` page using pure `buildAgeGroups`/`buildGroups` in `src/lib/coordinator/groups.ts` (tests in groups.test.ts). `buildGroups` adds size/count modes + a `mix` round-robin branch. Target size 10, `Math.ceil` group count so no group exceeds 10, balanced sizes (23→8/8/7, larger groups first), age-ordered youngest-first, ties by name. Age is best-available via `ageFor` (failsafe/print-data.ts): `age_at_registration` else derived from `dob`, else `null`. Groups operate on attending-only (status view filtered `attending=true`). **Known gap:** kids with neither dob nor age (`ageFor`→null) sort last but are still chunked into a group (render "age —"), not separated/flagged — the "must not silently land in a group" requirement is only partially met.

**Route builder (autoAssignStopsFromAddresses, routing.ts, admin client, coordinator-gated):** geocodes family addresses → assigns nearest stop to un-routed van kids, fills only empty legs the mode needs (never overrides manual). No-address/geocode-fail kids are counted `flagged`, never silently vanned. Operates only on EXISTING attending day-records — does NOT create a missing day-record, so the edit screen's no-day-record dead end (B2) is unchanged.

**Test coverage:** `splitName` (registration-schema), `buildAgeGroups` (groups.test.ts), `route-build`/`routing`/`failsafe-print-data`/`gps-freshness` helpers tested. Still NO test on any roster server action (updateStudent/updateStudentDayRecord/updateTodayStops/families.ts). See [[name-rule-and-edit-screen]].
